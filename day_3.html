Day 3: Inheritance & Polymorphism

Building on yesterday's foundations, today we explore two cornerstone principles of OOP that enable code reuse and flexibility: Inheritance and Polymorphism.

Base and Derived Classes in C#

Inheritance is a mechanism where a new class (the derived or child class) inherits properties and methods from an existing class (the base or parent class). This promotes code reusability and establishes a logical "is-a" relationship.

Analogy: Think of a Vehicle as a base class. It has properties like Speed and methods like Move(). A Car and a Bicycle can be derived classes. They both *are* vehicles, so they inherit the ability to move, but they might implement it differently or add their own unique features (like a Car having an Engine).

Syntax: Defining Inheritance

public class BaseClassName 
{ 
    // Members of the base class 
}

// The colon ':' indicates that DerivedClassName inherits from BaseClassName
public class DerivedClassName : BaseClassName 
{ 
    // Members of the derived class
}

Abstract Classes vs. Interfaces

Both are used to achieve abstraction, but they serve different purposes. An abstract class is a blueprint for related classes (an "is-a" relationship), while an interface defines a capability or contract that a class can implement (a "can-do" relationship).

Key Difference: A class can inherit from only one abstract class, but it can implement multiple interfaces. Think of it like a person: you can only have one biological father (single inheritance), but you can have multiple jobs or roles like 'Doctor', 'Parent', 'Musician' (multiple interfaces).

Method Overriding (virtual, override, sealed)

Method overriding allows a derived class to provide its own specific implementation of a method that is already defined in its base class. This is a key part of polymorphism.

  • virtual: Used in the base class to indicate that a method *can* be overridden by derived classes. It's an invitation to specialize.
  • override: Used in the derived class to provide a new implementation for a virtual method from the base class. It's accepting the invitation.
  • sealed: Used on an overridden method to prevent any further derived classes from overriding it again. It's ending the chain of specialization for that method.

Syntax: The Override Chain

public class Base
{
    public virtual void MyMethod() { /* Default behavior */ }
}
public class Derived : Base
{
    // Override provides a new behavior
    public sealed override void MyMethod() { /* New, final behavior */ }
}
public class FurtherDerived : Derived
{
    // COMPILE ERROR: Cannot override MyMethod because it is sealed
}

Polymorphism in Backend Practice

Polymorphism (from Greek, meaning "many forms") allows us to treat objects of different classes that share a common base class or interface as if they were objects of that base type. This simplifies code and makes it incredibly flexible.

Example Scenario: A payment processing system. You don't want to write separate logic for every single payment type. Instead, you can create a common PaymentMethod base class or interface with a ProcessPayment() method. Then, you create specific classes like CreditCardPayment and UpiPayment that each provide their own implementation of ProcessPayment().

public abstract class PaymentMethod { public abstract void Process(decimal amount); }
public class CreditCardPayment : PaymentMethod { public override void Process(decimal amount) => Console.WriteLine($"Processing credit card payment of ${amount}."); }
public class UpiPayment : PaymentMethod { public override void Process(decimal amount) => Console.WriteLine($"Processing UPI payment of ${amount}."); }

// Polymorphic usage
List<PaymentMethod> payments = new List<PaymentMethod>
{
    new CreditCardPayment(),
    new UpiPayment()
};

foreach (var paymentMethod in payments)
{
    // The correct Process method is called for each object type!
    paymentMethod.Process(100.50m);
}

Base: Shape Hierarchy

Create an abstract base class Shape with an abstract method CalculateArea(). Then, derive Circle and Rectangle classes from it and implement the area calculation for each.

Show Solution 
// C# Solution for Shape Hierarchy
using System;
using System.Collections.Generic;

public abstract class Shape { public abstract double CalculateArea(); }

public class Circle : Shape
{
    public double Radius { get; }
    public Circle(double radius) { Radius = radius; }
    public override double CalculateArea() => Math.PI * Radius * Radius;
}

public class Rectangle : Shape
{
    public double Width { get; }
    public double Height { get; }
    public Rectangle(double width, double height) { Width = width; Height = height; }
    public override double CalculateArea() => Width * Height;
}

[Easy] Animal Hierarchy

Implement a base class Animal with a virtual method MakeSound(). Create Dog and Cat subclasses that override this method to print "Woof" and "Meow" respectively.

Show Solution
// C# Solution for Animal Hierarchy
using System;
public class Animal { public virtual void MakeSound() => Console.WriteLine("Generic sound"); }
public class Dog : Animal { public override void MakeSound() => Console.WriteLine("Woof"); }
public class Cat : Animal { public override void MakeSound() => Console.WriteLine("Meow"); }

public class Program
{
    public static void Main()
    {
        var animals = new List<Animal> { new Animal(), new Dog(), new Cat() };
        foreach(var animal in animals) { animal.MakeSound(); }
    }
}

ðŸš— Case Study: Ride-Hailing System

Let's design a simplified model for a ride-hailing service. This is a classic example where polymorphism shines, allowing the system to handle different ride types through a common interface.

3. Polymorphism: The Key to Flexibility

We create an abstract Ride class. Then, each specific ride type inherits from it, providing its own fare logic:

  • EconomyRide: Standard fare (base + per km).
  • PremiumRide: Higher rates plus a surcharge.
  • RidePool: Fare is split among riders.

The system's core logic can work with any object of type Ride. When it calls CalculateFare(), polymorphism ensures the correct calculation is used for an Economy, Premium, or Pool ride without any `if/else` checks.

Show Full C# Implementation
// C# Solution for Ride-Hailing System
using System;
public class Rider { public string Name { get; set; } }
public class Driver { public string Name { get; set; } }

public abstract class Ride
{
    // ... Common properties ...
    public abstract decimal CalculateFare();
}

public class EconomyRide : Ride { /* ... Implementation ... */ }
public class PremiumRide : Ride { /* ... Implementation ... */ }
public class RidePool : Ride { /* ... Implementation ... */ }

public class RideHailingService
{
    // This method works with ANY object that is a 'Ride'
    public void CompleteRide(Ride ride)
    {
        // ... print details ...
        decimal fare = ride.CalculateFare(); // Polymorphism!
        Console.WriteLine($"Final Fare: ${fare:F2}");
    }
}
