SOLID Principles (Days 22–28)

Day 25: Interface Segregation Principle (ISP)

✅ Theory: Breaking Down the 'I' in SOLID

Definition

The Interface Segregation Principle (ISP) states that “Clients should not be forced to depend on interfaces they don’t use.”

In Simple Terms: It's better to have many small, specialized interfaces than one large, general-purpose one. Clients (classes) should only need to know about the methods that are relevant to them.

The Problem of "Fat" Interfaces

A "fat" or "polluted" interface is an interface with too many methods, covering a wide range of functionalities. When a class implements a fat interface, it's forced to provide implementations for all its methods, even those it doesn't need. This leads to several problems:

  • Unnecessary Implementation: Classes have to implement methods with empty bodies or by throwing exceptions like NotImplementedException, which is a code smell.
  • High Coupling: A change to one part of the interface (e.g., adding a new method) can force changes in all implementing classes, even those that don't use the new method.
  • Lower Cohesion: The interface bundles unrelated responsibilities, making the system harder to understand and maintain.

Real-World Analogy: The All-in-One Gadget

Imagine buying an all-in-one gadget that's a printer, scanner, fax machine, and coffee maker. This is a "fat interface."

Now, what if you only need a printer? You're still stuck with the complexity and cost of the other features. If the coffee maker part breaks, you might have to send the whole unit for repair, leaving you without a printer.

ISP suggests it's better to have separate, dedicated devices (a printer, a scanner, etc.). You only buy what you need, and a problem with one doesn't affect the others. This is the essence of splitting large interfaces into smaller, role-specific ones.

✅ Coding Practice

Let's refactor code to adhere to ISP using C#. These examples demonstrate how to break down large interfaces for different levels of complexity.

[Easy] The Multi-Function Machine

We start with the "fat" IMachine interface and refactor it into smaller, role-specific interfaces.

Before Refactoring (Problem)

// A "fat" interface
public interface IMachine
{
    void Print(string document);
    void Scan(string document);
    void Fax(string document);
}

public class SimplePrinter : IMachine
{
    public void Print(string document)
    {
        Console.WriteLine($"Printing {document}");
    }

    // Forced to implement methods it doesn't use
    public void Scan(string document)
    {
        throw new NotImplementedException("This machine cannot scan.");
    }

    public void Fax(string document)
    {
        throw new NotImplementedException("This machine cannot fax.");
    }
}
After Refactoring (Solution)

// Segregated interfaces
public interface IPrinter
{
    void Print(string document);
}

public interface IScanner
{
    void Scan(string document);
}

public interface IFaxer
{
    void Fax(string document);
}

// A class that can do all three
public class MultiFunctionPrinter : IPrinter, IScanner, IFaxer
{
    public void Print(string document) { /* ... */ }
    public void Scan(string document) { /* ... */ }
    public void Fax(string document) { /* ... */ }
}

// A simple printer only needs to implement IPrinter
public class EconomicPrinter : IPrinter
{
    public void Print(string document)
    {
        Console.WriteLine($"Printing {document} economically.");
    }
}

[Medium] Notification System

Consider a notification service. Not all notifications are sent via all channels. ISP helps us design this cleanly.

Problem: A Single Notification Interface

public interface INotifier
{
    void SendEmail(string to, string message);
    void SendSms(string phoneNumber, string message);
    void SendPushNotification(string deviceId, string message);
}

// This client only needs to send emails, but is forced to know about SMS and Push.
public class EmailNotifierClient : INotifier
{
    public void SendEmail(string to, string message) { /* ... */ }
    public void SendSms(string phoneNumber, string message) => throw new NotImplementedException();
    public void SendPushNotification(string deviceId, string message) => throw new NotImplementedException();
}
Solution: Segregated Sender Interfaces

public interface IEmailSender
{
    void SendEmail(string to, string message);
}

public interface ISmsSender
{
    void SendSms(string phoneNumber, string message);
}

// A notification service might depend on both
public class NotificationService
{
    private readonly IEmailSender _emailSender;
    private readonly ISmsSender _smsSender;

    public NotificationService(IEmailSender emailSender, ISmsSender smsSender)
    {
        _emailSender = emailSender;
        _smsSender = smsSender;
    }
    // ...
}

// A specific client only needs one
public class PasswordResetService
{
    private readonly IEmailSender _emailSender;

    public PasswordResetService(IEmailSender emailSender)
    {
        _emailSender = emailSender;
    }

    public void ResetPassword(string email)
    {
        _emailSender.SendEmail(email, "Your password reset link...");
    }
}

✅ Case Study: Printer System Design

Let's expand on the printer example to design a system with different types of devices. By applying ISP from the start, we create a flexible and maintainable system.

Step 1: Define Role-Specific Interfaces

Instead of one giant IDevice interface, we identify the core, independent functionalities.


public interface IPrinter { void Print(string content); }
public interface IScanner { void Scan(string content); }
public interface IFaxer { void Fax(string content); }
public interface IStapler { void Staple(int pages); }

Step 2: Implement Classes Based on Roles

Each class now implements only the interfaces for the features it actually provides. This leads to lean, focused classes.


// A high-end machine that does everything
public class AllInOneBeast : IPrinter, IScanner, IFaxer, IStapler
{
    public void Print(string content) => Console.WriteLine("Printing...");
    public void Scan(string content) => Console.WriteLine("Scanning...");
    public void Fax(string content) => Console.WriteLine("Faxing...");
    public void Staple(int pages) => Console.WriteLine("Stapling...");
}

// A simple home printer
public class BasicHomePrinter : IPrinter
{
    public void Print(string content) => Console.WriteLine("Printing from home...");
}

// A standalone scanner
public class PhotoScanner : IScanner
{
    public void Scan(string content) => Console.WriteLine("Scanning photo with high resolution...");
}

Step 3: Write Client Code That Depends on Abstractions

Our client code can now depend on the smallest possible interface it needs to do its job. For example, a "Print Service" only needs to know about IPrinter.


public class DocumentPrintService
{
    // This service only cares about printing. It doesn't know or care if the
    // machine can also scan, fax, or staple.
    public void PrintDocument(IPrinter printer, string document)
    {
        Console.WriteLine("--- Sending document to print service ---");
        printer.Print(document);
        Console.WriteLine("--- Print job complete ---");
    }
}

public static class Program
{
    public static void Main()
    {
        var printService = new DocumentPrintService();
        
        var allInOne = new AllInOneBeast();
        var homePrinter = new BasicHomePrinter();
        // var scanner = new PhotoScanner(); // Can't be used by PrintService!

        printService.PrintDocument(allInOne, "Annual Report");
        printService.PrintDocument(homePrinter, "Vacation Photos");
    }
}

Key Takeaway: The DocumentPrintService is decoupled from the specific type of printer. It can work with any object that implements IPrinter, making the system highly flexible and testable. This is the power of ISP combined with the Dependency Inversion Principle.

Test your understanding of the Interface Segregation Principle.

✅ Self-Assessment Challenge

Apply what you've learned to a new problem.

Scenario: A Role-Based API System

You are designing a backend system for a content platform. There are different user roles: User, Moderator, and Admin. You start with a single "fat" interface for all possible actions:


// The problematic "fat" interface
public interface IApiActions
{
    void ReadArticle(int articleId);     // For all roles
    void WriteComment(int articleId);    // For Users
    void DeleteComment(int commentId);   // For Moderators
    void BanUser(int userId);            // For Admins
}

Your Task: Refactor the IApiActions interface into smaller, segregated interfaces that better represent the different roles. Then, show how a class representing a controller for a simple User would implement the new interface(s).

Challenge Solution

The key is to create interfaces based on capabilities or roles. An action might even be shared across roles.


// Step 1: Segregate interfaces by role/capability

// Common action for anyone who can view content
public interface IArticleReader
{
    void ReadArticle(int articleId);
}

// Actions specific to a standard User
public interface IUserActions
{
    void WriteComment(int articleId);
}

// Actions specific to a Moderator
public interface IModeratorActions
{
    void DeleteComment(int commentId);
}

// Actions specific to an Admin
public interface IAdminActions
{
    void BanUser(int userId);
}


// Step 2: Implement a controller for a specific role

// The UserApiController only needs to handle reading and commenting.
// It implements only the interfaces relevant to its role.
public class UserApiController : IArticleReader, IUserActions
{
    public void ReadArticle(int articleId)
    {
        Console.WriteLine($"User is reading article {articleId}.");
    }

    public void WriteComment(int articleId)
    {
        Console.WriteLine($"User is writing a comment on article {articleId}.");
    }
}

// For comparison, an Admin controller would implement more interfaces
public class AdminApiController : IArticleReader, IUserActions, IModeratorActions, IAdminActions
{
    public void ReadArticle(int articleId) { /* ... */ }
    public void WriteComment(int articleId) { /* Admin can also be a user ... */ }
    public void DeleteComment(int commentId) { /* ... */ }
    public void BanUser(int userId) { /* ... */ }
}

Benefit: The UserApiController is now completely decoupled from moderator or admin actions. We can change admin capabilities (e.g., add DeleteArticle) without forcing any changes on the UserApiController, leading to a much more robust and maintainable system.
